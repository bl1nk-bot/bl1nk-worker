# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 0.0.x   | :white_check_mark: |
| < 0.0.1 | :x:                |

> โปรเจกต์ **bl1nk-worker** อยู่ในช่วง early release  
> แนะนำให้อัปเดตเป็นเวอร์ชันล่าสุดเสมอเพื่อรับการแก้ไขด้านความปลอดภัย

---

## Reporting a Vulnerability

หากคุณพบช่องโหว่ด้านความปลอดภัยในโปรเจกต์ **bl1nk-worker**  
กรุณาแจ้งผ่านช่องทาง **ส่วนตัว** ตามขั้นตอนด้านล่าง  
**อย่าเปิด public issue** จนกว่าจะได้รับการแก้ไขเรียบร้อยแล้ว

### ช่องทางแจ้ง

- 📧 ส่งอีเมลไปที่: **team@bl1nk.site**

### สิ่งที่ควรระบุในรายงาน

- เวอร์ชันของ `bl1nk-worker` ที่คุณใช้
- คำอธิบายช่องโหว่โดยละเอียด
- ขั้นตอนในการทำซ้ำ (reproduction steps)
- ผลกระทบที่อาจเกิดขึ้น
- ข้อเสนอแนะในการแก้ไข (ถ้ามี)

### ระยะเวลาตอบกลับ

- เราจะตอบกลับภายใน **5 วันทำการ** เพื่อยืนยันว่าได้รับรายงานแล้ว
- อัปเดตความคืบหน้าทุก **7-10 วัน** จนกว่าจะปิด issue
- หลังแก้ไขและออก release ใหม่ เราจะแจ้งกลับให้คุณทราบ

### กระบวนการดำเนินการ

1. ได้รับรายงาน → ยืนยันการรับ (ภายใน 5 วัน)
2. วิเคราะห์และประเมินความรุนแรง
3. พัฒนาและทดสอบแพตช์
4. ออก release ใหม่ + CVE advisory (ถ้าจำเป็น)
5. เปิดเผยข้อมูลหลังจากแพตช์เผยแพร่แล้ว

### การเปิดเผยข้อมูล

- เราจะให้เครดิตผู้รายงานใน release notes หากต้องการ
- รายละเอียดช่องโหว่จะถูกเผยแพร่หลังจากมีแพตช์แล้วเท่านั้น

## Security Best Practices for Deployers

เมื่อนำ `bl1nk-worker` ไปใช้ใน environment ของคุณ:

- 🔐 เก็บ **`GITHUB_TOKEN`** ใน Modal secret เท่านั้น ห้าม hardcode ในโค้ด
- 🔑 ใช้ **`webhook-secret`** ที่คาดเดายากเพื่อป้องกันการเรียก webhook โดยไม่ได้รับอนุญาต
- 🧹 หลีกเลี่ยงการ log token หรือ secret ลง output สาธารณะ
- 🕒 ควรเปลี่ยน secret อย่างสม่ำเสมอตามนโยบายองค์กรของคุณ
